# Executor V1 Spec

This is the first serious step toward the Percepta-style goal:

`program tokens -> transformer executes via next-trace-token prediction -> visible trace -> result`

The key design choice is:

**Do not train the model to jump straight from input to answer.**

Train it to emit a **canonical execution trace** for a tiny virtual machine.

## Goal

Build a small local transformer that can autoregressively execute a bounded
stack machine in the browser by predicting the next valid trace token.

V1 is intentionally narrow:

- small integer arithmetic
- bounded stack
- bounded memory
- simple branching
- exact deterministic traces

Sudoku is **not** the first task. Sudoku comes later as a program running on
top of this execution substrate.

## Non-goals

V1 does not attempt:

- free-form natural language programming
- one-shot `input -> final answer` reasoning
- full WebAssembly
- full Sudoku execution
- Percepta-style fast-path decoding yet

The first success condition is much smaller:

**Given a short VM program, the model should emit the exact next-step trace
until `HALT`.**

## VM surface

V1 uses a tiny stack machine with bounded state.

### State

- instruction pointer: `ip`
- stack pointer: `sp`
- stack: up to `8` values
- memory cells: `m0..m7`
- output buffer: small append-only list

### Value range

Use signed integers in the range:

- `-15 .. 15`

Represent them as discrete tokens:

- `v_n15 .. v_n1`
- `v_0`
- `v_1 .. v_15`

This keeps the vocabulary fixed and decoding constrained.

### Instruction set

V1 opcodes:

- `PUSH <value>`
- `POP`
- `DUP`
- `SWAP`
- `LOAD <mem>`
- `STORE <mem>`
- `ADD`
- `SUB`
- `MUL`
- `EQ`
- `LT`
- `JMP <addr>`
- `JZ <addr>`
- `OUT`
- `HALT`

This is enough to learn:

- stack mutation
- memory mutation
- control flow
- branching
- visible output

## Token vocabulary

The model vocabulary should be small, explicit, and non-natural-language.

### Structural tokens

- `<BOS>`
- `<PROGRAM>`
- `<TRACE>`
- `<STEP>`
- `<END>`
- `<NL>`

### Opcode tokens

- `PUSH`
- `POP`
- `DUP`
- `SWAP`
- `LOAD`
- `STORE`
- `ADD`
- `SUB`
- `MUL`
- `EQ`
- `LT`
- `JMP`
- `JZ`
- `OUT`
- `HALT`

### Address tokens

- `a00 .. a31`

Only a subset may be used in V1, but the token surface should be fixed.

### Memory tokens

- `m0 .. m7`

### Value tokens

- `v_n15 .. v_n1`
- `v_0`
- `v_1 .. v_15`

### Trace field tokens

- `ip`
- `op`
- `arg`
- `sp`
- `top`
- `mem`
- `out`
- `jmp`
- `taken`
- `not_taken`
- `halted`

## Program format

Programs are represented as explicit token lines.

Example:

```txt
<BOS>
<PROGRAM>
PUSH v_3
PUSH v_5
ADD
OUT
HALT
<TRACE>
```

The model starts decoding after `<TRACE>`.

## Canonical trace format

Each instruction execution must map to exactly one canonical trace line.

### Trace line schema

```txt
<STEP> ip a02 op ADD arg _ sp v_1 top v_8 mem _ out _
```

Rules:

- one executed instruction = one trace line
- always emit fields in the same order
- use `_` when a field does not apply
- branch instructions emit whether the jump was taken
- `OUT` emits the appended value
- `HALT` emits `halted`

### Canonical examples

Program:

```txt
PUSH v_3
PUSH v_5
ADD
OUT
HALT
```

Trace:

```txt
<STEP> ip a00 op PUSH arg v_3 sp v_1 top v_3 mem _ out _
<STEP> ip a01 op PUSH arg v_5 sp v_2 top v_5 mem _ out _
<STEP> ip a02 op ADD arg _ sp v_1 top v_8 mem _ out _
<STEP> ip a03 op OUT arg _ sp v_0 top _ mem _ out v_8
<STEP> ip a04 op HALT arg _ sp v_0 top _ mem _ out halted
<END>
```

The trace must be deterministic. That matters more than being pretty.

## Data format

Training data should be generated by a deterministic interpreter, not handwritten.

### JSONL row

```json
{
  "id": "add_000001",
  "program": [
    "PUSH v_3",
    "PUSH v_5",
    "ADD",
    "OUT",
    "HALT"
  ],
  "trace": [
    "<STEP> ip a00 op PUSH arg v_3 sp v_1 top v_3 mem _ out _",
    "<STEP> ip a01 op PUSH arg v_5 sp v_2 top v_5 mem _ out _",
    "<STEP> ip a02 op ADD arg _ sp v_1 top v_8 mem _ out _",
    "<STEP> ip a03 op OUT arg _ sp v_0 top _ mem _ out v_8",
    "<STEP> ip a04 op HALT arg _ sp v_0 top _ mem _ out halted",
    "<END>"
  ],
  "metadata": {
    "family": "add_two",
    "steps": 5
  }
}
```

### Program families

Start with synthetic families:

- add two numbers
- subtract two numbers
- multiply two numbers
- duplicate / swap exercises
- load / store exercises
- branch-if-zero exercises
- tiny counted loops

The point is not diversity for its own sake. The point is to cover VM semantics
systematically.

## Training objective

V1 training objective:

- causal next-token prediction over the concatenated `program + trace`

Input:

```txt
<BOS>
<PROGRAM>
...
<TRACE>
```

Target:

- next trace token until `<END>`

Important constraint:

**The trace must be canonical.**

If multiple traces can represent the same execution, training quality collapses.

## Constrained decoding

Even before fast-path attention, decoding should be constrained.

Given the current partial line, only allow valid next-token classes:

- after `ip` -> address token
- after `op PUSH` -> value token
- after `op LOAD` -> memory token
- after `op ADD` -> `arg _`
- after `out` -> `_`, a value token, or `halted`

This does two things:

- reduces invalid traces
- makes the executor feel much more computer-like

## Browser runtime plan

V1 browser architecture:

- main thread: UI only
- worker: decode loop
- optional second worker later for verification or comparison

### Worker message protocol

Request:

```json
{
  "type": "run_program",
  "program": [
    "PUSH v_3",
    "PUSH v_5",
    "ADD",
    "OUT",
    "HALT"
  ]
}
```

Streamed events:

```json
{ "type": "token", "value": "<STEP>" }
{ "type": "token", "value": "ip" }
{ "type": "token", "value": "a00" }
{ "type": "line", "value": "<STEP> ip a00 op PUSH arg v_3 sp v_1 top v_3 mem _ out _" }
{ "type": "done", "steps": 5 }
```

Support:

- `load_model`
- `run_program`
- `cancel`
- `reset`

## Path to Sudoku

Sudoku should enter only after the VM can handle:

- memory
- branching
- repeated state updates
- longer traces

The staged path is:

### Stage A

Compile simple arithmetic DSL programs into VM code.

### Stage B

Compile small search tasks:

- 24 game
- tiny path finding
- 4x4 Latin square

### Stage C

Compile 4x4 Sudoku into VM code.

### Stage D

Move to constrained 9x9 Sudoku traces.

At that point the transformer is no longer solving Sudoku as a one-shot guess.
It is executing a compiled search procedure through its own decoded trace.

That is the right benchmark for the "computer inside transformer" direction.

## Why this is the right first build

This spec gives us:

- exact supervision
- infinite synthetic data
- small vocabulary
- clear correctness checks
- a browser-friendly local model target

Most importantly, it changes the problem from:

- `Can a model guess Sudoku?`

to:

- `Can a transformer learn to execute a bounded machine trace exactly?`

That is a much stronger foundation.
