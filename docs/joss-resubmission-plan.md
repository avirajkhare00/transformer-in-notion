# JOSS Resubmission Plan

## Why JOSS was not a fit yet

The first JOSS submission was rejected at pre-review because the repository's
public development history was too new. The software package itself was not the
main problem. The practical takeaway is that a later JOSS resubmission should
be treated as a public-history milestone, not as a packaging milestone.

## Target timing

- earliest realistic re-evaluation: September 2026
- better target: after the repository shows multiple public releases and normal
  issue or pull-request activity over time

## What should happen before resubmission

1. keep development in the public repository instead of only in local branches
2. ship additional tagged releases with clear release notes
3. use public issues for roadmap and bug tracking
4. keep the JOSS paper synchronized with the software artifact
5. preserve a clean archival path for the chosen submission release if a DOI is minted later

## Repository signals worth building

- recurring releases
- visible issue discussion
- bug fixes tied to public issues
- external reproducibility or usage feedback, if any appears
- continued tests and documentation maintenance

## Submission strategy

- use arXiv now for the broader PSVM research paper
- keep the JOSS paper as the software-focused artifact paper
- resubmit to JOSS only after the repository has aged enough in public
